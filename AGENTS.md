# AWM Baseline Registry — Agent Notes

Lecciones de estilo de trabajo y patrones que funcionan bien, agnósticas de qué modelo/agente esté operando en este repo.

## What works here

- **Al cerrar una rama de contenido contra `main`, el bump de versión de bundle (metadata en `catalog.json` + `bundles/<x>/bundle.json` + frontmatter del `SKILL.md`) es tu responsabilidad en el PR; el tag `vX.Y.Z` lo corta la CI sola.** `.github/workflows/auto-tag.yml` bumpea el último tag semver al mergear a `main` (nivel por conventional commits del merge). Obligación de release codificada en `CONSTITUTION.md` ("Release de contenido"). **Ojo — son dos versiones independientes:** el tag es la versión del registry (lo único que `awm update` consume: pin > último tag > HEAD); las `version` de bundle son metadata que NO determina el número del tag (en `v1.4.0`, `dev`=1.4.0 y `frontend`=2.0.0). **Cuándo aplica:** cualquier PR que edite skills/prompts/sensor-packs de un bundle — incluí el bump de `catalog.json` y `bundle.json` en el mismo PR; el tag sale automático al mergear (fallback manual si CI falla, ver CONSTITUTION). **El bump del propio `SKILL.md` editado (frontmatter `version`) ya no depende solo de acordarte:** `.github/workflows/skill-version-check.yml` lo falla en CI si el contenido cambió y la versión no (harness-retro 2026-07-23, tras 5+ ocurrencias de este mismo olvido en una sola sesión) — corré `./scripts/check-skill-version-bumps.sh origin/main` localmente antes de empujar si querés adelantarte al check. **Titulá el PR como conventional commit** (`feat:`/`fix:`/`feat!:`) — el squash-merge (default de este repo) usa ese título tal cual como subject del commit en `main`, y el auto-tag deriva el nivel de bump de ese subject; un título descriptivo sin prefijo cae en `patch` sin importar la severidad real (pasó en el PR #13: retiro de una skill entera tageado como patch).

- **Al agregar contenido nuevo a un archivo que ya tiene secciones de reglas (Anti-patterns, Red Flags, Common Mistakes), revisar proactivamente si el contenido nuevo contradice alguna regla existente — antes de reportar DONE, no después de que un reviewer lo encuentre.** Confirmado: en una tarea que añadía un carve-out de modo a `finishing-a-development-branch/SKILL.md`, el implementer leyó las secciones "Common Mistakes"/"Red Flags" existentes por su cuenta, encontró que dos líneas ("Fix: Present exactly 4 structured options", "Always: Present exactly 4 options") quedarían contradichas por el cambio, y las corrigió en el mismo commit — evitando el ciclo extra de fix-and-rereview que sí ocurrió en tareas hermanas (`harness-retro/SKILL.md`, `subagent-driven-development/SKILL.md`) donde la misma clase de contradicción solo se detectó en la revisión de calidad de código, ya con el commit hecho. **Cuándo aplica:** cualquier edición que agregue una excepción/carve-out a un archivo con secciones normativas propias (reglas, anti-patrones, invariantes) — leer esas secciones enteras antes de reportar terminado.

- **Al marcar una skill `portable: true` (exportable a claude.ai vía `awm export`), verificar que su body sea self-contained en el sentido fuerte: cualquier path intra-registry que cite (`skills/<x>/SKILL.md`, `skills/<x>/references/<y>.md`) apunta a contenido que NO viaja en el export y queda muerto en claude.ai.** Si la skill DEFIERE contenido funcional a un archivo de otra skill (ej. `product-brief` deriva el contrato del brief de `skills/readiness-gate/references/brief-contract.md`), necesita un override `skills/<name>/port.claude-ai.md` que reproduzca ese contenido inline — sin él, el port apunta al vacío (esto motivó el override de `product-brief`, 2026-07-23). Si solo son menciones blandas (el contenido ya está inline justo después del path, o la skill referenciada se exporta al lado), la skill FUNCIONA standalone pero el export mecánico igual embarca el path muerto — cosmético, no bloqueante. **No lo cures con un override per-skill solo para limpiar paths cosméticos:** un override es un duplicado del body que reintroduce el sync manual que `awm export` existe para eliminar, y editar el SKILL.md canónico degradaría Claude Code (donde esos paths SÍ resuelven). El fix del ruido cosmético pertenece al transform mecánico de la herramienta ([agentic-workflow#12](https://github.com/Kodria/agentic-workflow/issues/12)); el override se reserva para diferencias funcionales reales (contrato inline), no para path-cleanup. **Cuándo aplica:** todo PR que agregue `portable: true` a una skill — leé su body y clasificá cada path intra-registry como dependencia dura (necesita override) o mención blanda (funciona, el ruido lo limpia el tool).

- **Tras completar todas las tareas por-tarea de un plan multi-fase (cada una con su propia revisión spec+calidad), correr una revisión final de "sistema completo" que trace journeys de usuario end-to-end sobre los archivos reales — no una tarea más del mismo tipo.** Confirmado en una sesión de 12 tareas: las 11 revisiones por-tarea (cada una comparando su propio diff contra el plan) pasaron limpias en su mayoría, pero una revisión final dedicada a trazar 2 recorridos completos (idea cruda → discovery → brief → gate → handoff → dev; repo existente → extraction → brief → mismo camino) encontró 5 hallazgos reales que ninguna revisión individual podía ver — incluyendo un journey completo que fallaba porque un handoff prometido por un skill no tenía mecanismo de recepción en el otro. Las revisiones por-tarea miden fidelidad contra el plan; la revisión de sistema completo mide si las piezas, ya construidas, realmente encajan entre sí. **Cuándo aplica:** cualquier plan con ≥2 tareas cuyo resultado se invoca entre sí (un orquestador + los skills que rutea, un contrato compartido + sus consumidores) — la revisión final de sistema completo no es opcional aunque cada tarea individual haya pasado limpia. Ver también CONSTITUTION.md ("Revisión de código" → re-verificar artefactos compartidos mutables contra su estado actual).
